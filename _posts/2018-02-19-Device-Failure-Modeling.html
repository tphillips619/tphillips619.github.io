end_dates3 = []

for dev in unique_devices:
    # Create a device dataframe for each unique device
    device_df = df[df['device']==dev]
    
    start_dates.append(device_df.iloc[0]['date'])
    end_dates.append(device_df.iloc[-1]['date'])
    
    # Find second to last end date
    end_dates2.append(device_df.iloc[-2]['date'])
    odometer_diff2.append(device_df.iloc[-1]['mileage']-device_df.iloc[-2]['mileage'])
    
    # Find Nth to last end date
    nth_index = -(last_N_days+1)
    if (len(device_df)<last_N_days):                 # If N days is larger that the number of days in the devices' observations
        end_dates3.append(device_df.iloc[0]['date']) #  then use the first date in the series
        odometer_diff3.append(device_df.iloc[-1]['mileage']-device_df.iloc[0]['mileage'])
    else:
        end_dates3.append(device_df.iloc[-3]['date'])
        odometer_diff3.append(device_df.iloc[-1]['mileage']-device_df.iloc[nth_index]['mileage'])

df_unique_devices = pd.DataFrame(start_dates, index=unique_devices, columns=['start_date'])

Downsampling majority class¶

Create a subset dataframe that only includes the last date for each device. This ignores the time dimension and attempts to build a model by only comparing devices that stopped recording without failing, and devices that stopped recording when they failed. This will predict devices that are prone to immediate failure.

Feature Engineering¶

Create new feature 'day'¶

Creating a new feature 'day' as an integer of the number of days since 2015-01-01. Since failed devices do not come back, the new 'day' feature is a number of consecutive days without failure. The state of the devices prior to the recorded observations is not known, and there could be many more days without failure prior to the start of the recorded observations. However, devices with a larger 'day' feature should have an increasingly higher probability of failing.

One Hot Encoding: Creating Dummy Variables¶

The lack of information available to describe this dataset requires that I make educated guesses about whether each feature is a numerical or categorical variable. From the data exploration earlier, I noticed that for most of the feature columns, only a few observations have non-zero values. I'll make an educated guess that these non-zero values are important in identifying devices that will fail. At the risk of overfitting I'm going to use a one-hot encoding scheme to treat most columns. I'll review bias/variance tradeoffs when evaluating the model and favor shallow trees to avoid overfitting.

Option to Upsample Minority Class¶

The pervasive challenge with this dataset is the imbalance between positive and negative classes and the small number of positive observations. One technique to address this is to upsample the minority class with synthetic data points that are similar, but not identical to, real datapoints. This must be done after the train/test split and only on the training data. The model should only predict on real observations.

Although its possible to use Synthetic Minority Over-sampling Technique (SMOTE) to address the imbalance, I'm going to ignore this for now and only revisit it if my current strategy doesn't work.

There are at least 10 negative cases for every 1 positive case. At the suggestion of the xgboost parameter documentation, a 'scale_pos_weight' of 10 should be considered. I'll run the gridsearch with a 'scale_pos_weight' of 1 and 10.

When conducting a gridsearch of hyperparameters to use with the model, Sklearn's metric Average Precision (AP) summarizes both Precision and Recall by using the weighted mean of precisions achieved at each threshold, with the increase in recall from the previous threshold used as the weight. Setting the parameter scoring='average_precision' when conducting GridSearchCV will identify a model that maximizes Precision and Recall over each cross-validated fold.

Choosing a Metric to Evaluate the Model¶

$$ Accuracy = \frac{TP+TN}{TP+TN+FP+FN}$$$$ Recall = \frac{TP}{TP+FN}$$$$ Precision = \frac{TP}{TP+FP}$$

There are 10 working devices for every failed device. If Accuracy is used to evaluate a model, all models will have an easy time attaining a high accuracy score by producing predictions in which it predicts most devices will remain operational. The model we want is one which maximizes Precision and Recall.

Precision measures the percentage of predictions that the model gets right. A model with high precision will rarely pull a device from service unless it will fail. Recall measures the percentage of device failures that the model catches. A model with a high recall score indicates that the model will rarely fail to flag a device that will fail.

A simple strategy to reduce overfitting is to use information gleaned from evaluating feature importances from this iteration to select better features in future iterations. The next iteration of this model would involve combining non-zero one-hot encoded attributes and letting the algorithm find an appropriate splitting criteria. I would also experiment with adding additional restraints on the shape of the trees such as specifying a minimum number of samples that a leaf node must have or the minimum number of a samples that a decision node must have in order to split into leaf nodes.

The pervasive challenge of this dataset is the imbalance between positive and negative classes and the rare occurence of positive classes. It was a challenge to create a model that does not always classify each observation as the majority class. Creating a model that was capable of predicting the occurence of rare events involved:

  1. Downsampling the majority class
  2. Selecting a model type that is robust to class imbalances (a decision tree based model)
  3. Adding a penalty term to misclassification of the minority class ('scale_pos_weight'=10)
  4. Choosing a scoring metric that utilized Precision and Recall when selecting hyper-parameters for the model (scoring='average_precision')
